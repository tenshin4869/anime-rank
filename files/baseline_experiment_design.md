# ベースライン実験設計書
## Phase 0 — フラット学習による人気予測の可能性と限界の検証

---

## 1. 実験の位置づけと目的

### 背景（指導教員FBへの応答）

指導教員より「コア/ライト層の分離なしに全特徴量をフラットに学習させれば、MLが自動的に人気予測できるのではないか」という問いが提起された。この問いに対して、研究の主張（コア/ライト分離の必要性）を仮説の出発点とするのではなく、**実験から導出される結論として示す**ことが本実験の位置づけである。

### 実験の3つの目的

1. **フラット学習で何が起きるかを定量的に確認する**
   - ベースライン精度の計測（うまくいく部分の確認）
   - 予測が崩れる条件・作品群の特定（うまくいかない部分の確認）

2. **どの特徴量が仕事をしているか・していないかを特定する**
   - 特徴量重要度の分析（SHAP等）
   - 特定の特徴量が消えてしまっている現象の発見

3. **コア/ライト分離が必要な根拠をデータで示す**
   - 「分けなくていい」を否定するのではなく、「分けないと何が見えなくなるか」を具体的に示す

## 2. タスク定義

### 予測タスクの概要

| 項目 | 設定 |
|------|------|
| ドメイン | アニメ（第1フェーズ） |
| サンプル数 | 50〜100作品 |
| 入力ウィンドウ | 放送開始後 第1〜4週 のデータ |
| 目的変数 (Y) | Google Trends スコア（0〜100、日次） |
| 予測タイミング | 複数時点（後述） |
| タスク種別 | 回帰（スコア予測） |

### 予測ターゲット（Yの時点）— 3時点

| ラベル | 時点の定義 | 意味 |
|--------|-----------|------|
| Y_mid | 放送中盤（放送開始から約6週後） | 作品が継続的に話題になっているか |
| Y_end | 放送終了直後（最終話から1週以内） | ピーク人気の規模 |
| Y_3m | 放送終了から3ヶ月後 | 定着人気・長期記憶の有無 |

## 3. データセット設計

### 対象作品の選定条件

- 2018年〜2023年に放送開始のTVアニメ（季節クールもの）
- Wikipediaの日本語ページが存在すること（データ取得の前提条件）
- 全話数が判明していること（放送中断・未完作品は除外）
- 1クール（12〜13話）または2クール（24〜26話）作品を中心に選定
- 超人気作品（鬼滅、進撃等）は含める（外れ値として分析時に別途確認）

### サンプリング方針

- MAL（MyAnimeList）スコアの分布を参考に、高・中・低人気を均等にサンプリングする
- 特定ジャンル偏りを避けるため、少年・少女・SF・日常系等を混在させる
- 目標：50作品（最小構成）→ 100作品（拡張）

## 4. 入力特徴量

**使用する特徴量は以下の4つのみ。**

| # | 特徴量名 | ソース | 粒度 |
|---|---------|--------|------|
| 1 | Wikipedia ページビュー数（PV） | Wikipedia Pageviews API | 日次 |
| 2 | Google Trends スコア | Google Trends (pytrends) | 日次 |
| 3 | Wikipedia 編集回数 | Wikipedia API (revisions) | 日次 |
| 4 | Wikipedia 編集者数（ユニーク） | Wikipedia API (revisions) | 日次 |

### タイムスタンプの統一

**4つの特徴量はすべて同一の日付（YYYY-MM-DD）を基準に対応させる。**

Wikipedia APIは日次データを返す。Google Trendsも日次粒度で取得し、同一日付で4特徴量が揃うようにする。取得後は日付をキーとして結合（LEFT JOIN）する。

---

## 5. Google Trends のデータ仕様と取得方針

### データの形式

Google Trendsから取得できるのは**検索人気の相対スコア（0〜100）の時系列**である。スコアの基準は**そのリクエスト期間内での最大検索量の日=100**として正規化される。

### 自己完結型の取得方針

研究タスクは「各作品の将来トレンド推移を予測する」ことであり、作品間のスコア比較は不要である。そのため**1作品につき1リクエスト**で入力期間〜予測時点までをまとめて取得する。同一リクエスト内でXとYは同じスケールで対応するため、正規化・アンカー補正は不要。

```
リクエスト：[Wikipediaタイトル] / JP / 放送開始日〜放送終了3ヶ月後

  Day 1〜28のスコア   → 入力X（第1〜4週）
  約Day 42のスコア    → Y_mid
  最終話翌週のスコア  → Y_end
  終了3ヶ月後のスコア → Y_3m
```

### 検索クエリの統一ルール

**Google TrendsのクエリはWikipedia日本語ページのタイトルに統一する。**

Wikipedia のページタイトルはその作品を代表する公式表記として管理されており、全作品で一貫したルールが存在する。独自のクエリ選定ロジックは持たない。

```
例：
  Wikipedia タイトル「チェンソーマン_(アニメ)」
  → クエリ：「チェンソーマン」（括弧内の補足表記を除去）

  Wikipedia タイトル「鬼滅の刃_(アニメ)」
  → クエリ：「鬼滅の刃」
```

**括弧表記の除去ルール：**
Wikipediaのタイトルには曖昧さ回避のために `_(アニメ)` や `_(テレビアニメ)` が付与されることがある。Google Trendsのクエリとして使用する際はこの括弧部分を除去する。除去後のクエリをmetadata.jsonに記録する。

### データ管理ルール

- **rawフォルダは編集禁止。** 前処理はprocessedフォルダにのみ書き込む
- **取得日時を必ず記録する。** Google Trendsはリクエスト時期によってスコアが変わる可能性があるため `data_collected_at` を必ず記載する
- **0とNaNを区別する。** PV=0はその日の実測値0、NaNはページが存在しない日（ページ未作成等）
- **作品名はスネークケース英語で統一する。** 日本語タイトルはmetadata.jsonの `anime_title_ja` に記録する
- **保存するCSVには生の値のみ記録する。** log1p等のスケーリングはモデル学習コード側でのみ適用する

## 7. 前処理

### PV=0 の扱い

- **PV=0は有効なデータとして保持する**（その日に閲覧がなかった事実を記録）
- **除外するのはページが存在しない日（NaN）のみ。** 放送開始時点でWikipediaページが存在しない作品はデータセットから除外する

### スケーリング（モデル学習時のみ適用）

| 特徴量 | 処理 | 理由 |
|--------|------|------|
| PV数 | log1p変換 | 桁数が大きくスケールが不均一になるため |
| Google Trendsスコア | そのまま | 0〜100の範囲に収まっているため |
| 編集回数・編集者数 | そのまま（必要に応じてlog1p） | 値が小さいため |

## 8. モデル設計

### 使用モデル

| モデル | 目的 |
|--------|------|
| 線形回帰（Ridge） | ベースライン・特徴量係数の直接確認 |
| Random Forest | 非線形関係の捕捉・特徴量重要度（MDI）の取得 |
| XGBoost | 精度上限の確認・SHAP値による解釈 |

ディープラーニングはこの段階では使わない（解釈性が低く、フラット学習の挙動観察に不向き）。

### 評価指標

| 指標 | 理由 |
|------|------|
| RMSE | 予測誤差の絶対量を把握 |
| Spearman 順位相関係数 ρ | 保住ら（2014）との比較可能性を確保 |
| R² | 説明力の全体像 |

3予測時点（Y_mid, Y_end, Y_3m）それぞれについて全指標を計測する。

### 検証方法

- 5-fold 交差検証（サンプル数が少ないため）
- 放送年で分割した時系列Split（2018〜2021を訓練、2022〜2023をテスト）を追加で確認

## 9. 分析計画

### Step 1：ベースライン精度の確認

各モデル・各予測時点でのRMSE, Spearman ρ, R² を表形式で整理する。残差が大きい作品群を列挙する。

### Step 2：特徴量重要度の分析

| 確認事項 | 期待する発見 |
|----------|-------------|
| PV と編集数の重要度の差 | 受動的信号と能動的信号で貢献が異なるか |
| Google Trends の自己予測力 | 初期スコアで将来スコアをどこまで予測できるか |
| 編集者数 vs 編集回数 | 人数と頻度のどちらが予測力を持つか |

### Step 3：失敗パターンの分類

- **仮説A（コア向け作品の過小評価）**：PV少・編集多の作品でTrendsを過小予測していないか
- **仮説B（バズり型作品の過大評価）**：初期Trends急上昇後に失速した作品でY_3mを過大予測していないか
- **仮説C（予測時点による精度差）**：Y_mid > Y_end > Y_3m の順で精度が落ちるか

### Step 4：消えている特徴量の特定

重要度が低いにもかかわらず理論的に重要なはずの特徴量を特定し、消える理由を考察する。この考察が次フェーズ（特徴量エンジニアリング）の設計根拠になる。

## 10. 成功・失敗の判断基準

| 結果パターン | 意味 | 次のアクション |
|-------------|------|---------------|
| Spearman ρ > 0.6 | 基本的な信号の有効性は確認できた | 精度の天井と失敗パターンを分析し、コア/ライト信号混在問題を示す |
| Spearman ρ < 0.4 | フラット学習の限界を示せた | 特徴量重要度から「何が足りないか」を特定し、コア/ライト分離の動機とする |
| Y_3m だけ精度が低い | 初期データに長期人気の信号が不足 | コア層信号（編集者数等）と長期人気の相関を次フェーズで検証 |
