## 研究の概要

**テーマ：** ドメイン横断コンテンツ人気予測フレームワーク

**研究課題（一言）：** アニメ・映画・音楽・ゲームなど異なるドメインのコンテンツについて、リリース直後の少ないデータから将来の人気を予測できる、単一の汎用フレームワークを構築できるか？

**ターゲットユーザー（誰のための予測か）：** コンテンツホルダー・二次利用事業者（放映権交渉、グッズ展開、配信優先度判断など）

---

## 重要な前提（誤解しやすいポイント）

- **推薦システムではない。** ユーザーに何かを薦めるシステムではなく、コンテンツそのものの社会的な人気規模を予測する
- **アニメは例示ドメインにすぎない。** 最終的なゴールはドメイン横断の汎用性。アニメ特有の処理や特徴量は原則として使わない
- **LLMは主役ではない。** 問題構造の設計が先で、LLMはあくまでデータ補完や特徴変換の手段として後から検討する
- **作品間のスコア比較は不要。** 各作品の将来トレンド推移を予測することが目的。「A作品とB作品どちらが人気か」を問うタスクではない

---

## 確定している研究の3つのコア主張

1. **シグナルバイアスの低減**
   - 能動的行動（Wikipedia編集）= コアファン層のシグナル
   - 受動的行動（ページビュー、検索）= ライトユーザー層のシグナル
   - どちらか一方だけに依存すると予測が歪む。両方を組み合わせることでバイアスを低減する

2. **クロスドメイン汎用性**
   - 同一フレームワーク・同一特徴量設計を複数ドメインで動かす
   - ゼロショット転移ではなく「共通構造 + ドメイン別データ」の設計

3. **スパースな初期データからの反復更新**
   - リリース直後の少ないデータから予測を開始し、データが増えるにつれて予測を更新する（オンライン学習的アプローチ）

## 現在の実験フェーズ：Phase 0（ベースライン実験）

### このフェーズの目的

指導教員のFBに応答する実験。「コア/ライト層を分けなくてもMLが自動的に人気予測できるのでは？」という問いに対して、フラットに全特徴量を学習させたときに何が起きるかを定量的に確認する。

**重要：このフェーズで高精度を出すことは目的ではない。失敗パターンと機能しない特徴量を特定することが目的。**

### タスク定義

```
入力X：放送開始後 Day1〜Day28（第1〜4週）の日次データ
         - Wikipedia PV数
         - Google Trends スコア
         - Wikipedia 編集回数
         - Wikipedia 編集者数（ユニーク）

出力Y：同一作品の将来時点のGoogle Trendsスコア
         - Y_mid：放送中盤（約6週後）
         - Y_end：放送終了直後
         - Y_3m ：放送終了3ヶ月後
```

### ドメインとサンプル

- ドメイン：アニメ（TVアニメ、2018〜2023年放送開始）
- サンプル数：50〜100作品
- サンプリング：MALスコアで高・中・低人気を均等に

## データ設計

### Google Trendsの取得方針（重要）

**自己完結型取得：** 1作品につき1リクエストで入力期間〜予測時点をまとめて取得する。同一リクエスト内でXとYは同じスケールで対応するため、アンカー正規化・作品間補正は一切不要。

**クエリの統一ルール：**
Google TrendsのクエリはWikipedia日本語ページのタイトルに統一する。Wikipediaのページタイトルはその作品を代表する公式表記として管理されており、全作品で一貫したルールが存在するため。

```
処理ルール：
  Wikipediaタイトル「チェンソーマン」
  → Google Trendsクエリ：「チェンソーマン」

  Wikipediaタイトル「鬼滅の刃」
  → Google Trendsクエリ：「鬼滅の刃」
```

採用したクエリは必ず `metadata.json` の `gt_query` フィールドに記録する。

### フォルダ構成

```
data/
├── raw/                  # 生データ（編集禁止・読み取り専用）
│   └── {anime_name}/
│       ├── wikipedia_daily.csv
│       ├── google_trends_daily.csv
│       └── metadata.json
├── processed/
│   ├── daily/            # {anime_name}_daily.csv
│   ├── weekly/           # {anime_name}_weekly.csv（週=7日間の総和）
│   └── master.csv        # 全作品フラット（モデル学習用・1行1作品）
└── labels/
    └── targets.csv       # Y_mid, Y_end, Y_3m の正解ラベル（全作品）
```

### データルール

| ルール | 詳細 |
|--------|------|
| raw/は読み取り専用 | 書き込みは processed/ のみ。既存ファイルの上書きは `--force` フラグ指定時のみ |
| 0とNaNを区別する | PV=0はその日の実測値0。NaN=ページ未存在（→その作品をデータセットから除外） |
| 生の値のみCSVに保存 | log1p等のスケーリングはモデルコード側でのみ適用する |
| 作品名はスネークケース英語 | 例：`chainsaw_man`。日本語名はmetadata.jsonの `anime_title_ja` に記録 |
| 取得日時を必ず記録 | metadata.jsonの `data_collected_at` に記載。GTスコアはリクエスト時期で変わる可能性がある |

### 日次ファイルのカラム構成（`{anime_name}_daily.csv`）

```
date, pv, gt_score, edit_count, editor_count, day_index, week_index, phase
```

- `phase` の値：`input` / `Y_mid` / `Y_end` / `Y_3m`
- `pv=0` は有効値。`pv=NaN` はページ未存在

### 週次ファイルのカラム構成（`{anime_name}_weekly.csv`）

```
week_start, week_end, week_index, pv_sum, gt_sum, edit_sum, editor_sum, phase
```

- 週の値はすべて7日間の**総和**

### metadata.jsonの必須フィールド

```json
{
  "anime_title":           "chainsaw_man",
  "anime_title_ja":        "チェンソーマン",
  "wikipedia_page_title":  "チェンソーマン_(アニメ)",
  "gt_query":              "チェンソーマン",
  "gt_query_rule":         "Wikipediaページタイトルから括弧表記を除去",
  "gt_request_period":     "2022-10-01_to_2023-03-31",
  "air_start":             "2022-10-11",
  "air_end":               "2022-12-27",
  "total_episodes":        12,
  "data_collected_at":     "2026-04-16",
  "wikipedia_page_exists": true,
  "notes":                 ""
}
```

## モデル設計（Phase 0）

### 使用モデル

| モデル | 用途 |
|--------|------|
| Ridge回帰 | ベースライン・係数の直接確認 |
| Random Forest | 特徴量重要度（MDI）の取得 |
| XGBoost | 精度上限の確認・SHAP値による解釈 |

ディープラーニングはPhase 0では使わない。

### 評価指標

RMSE, Spearman順位相関係数ρ, R² を3予測時点（Y_mid, Y_end, Y_3m）それぞれで計測する。

### 検証

5-fold交差検証 + 放送年による時系列Split（2018〜2021訓練、2022〜2023テスト）

## よくある作業リクエストと注意点

**データ収集スクリプトを書くとき**
→ `raw/` フォルダへの書き込みのみ行う。既存ファイルがある場合は上書きせず、`--force` フラグで明示指定された場合のみ上書きする

**特徴量生成スクリプトを書くとき**
→ `raw/` を読み込み、`processed/` に書き出す。スケーリングはここでは適用しない（生の値のみ保存）

**モデル学習スクリプトを書くとき**
→ `master.csv` を読み込む。スケーリングをここで適用する。Y_mid, Y_end, Y_3m をそれぞれ別のターゲットとして扱う

**分析スクリプトを書くとき**
→ 残差の大きい作品をCSVで出力する。SHAP値の可視化はmatplotlibで画像ファイルに保存する（`plt.show()` は使わない）

## 先行研究との対応

| 研究 | 本研究との関係 |
|------|--------------|
| 保住ら（2014） | 検索数・ツイート数・Wiki編集数の組み合わせで消費トレンド予測。評価指標（Spearman ρ）を合わせて比較可能にする |
| Asur & Huberman（2010） | Twitterによる映画興行収入予測。本研究ではTwitterは使わないがベースラインの考え方を参照 |
| Bass model（1969） | コア層（イノベーター）とライト層（フォロワー）の2段階普及モデル。コア/ライト分離の理論的背景として引用 |
